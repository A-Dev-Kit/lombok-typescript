# The `lombok-ts` CLI

Once you've installed `lombok-typescript` the `lombok-ts` binary lands in `node_modules/.bin/`. Invoke it via `npx lombok-ts` or add scripts to your `package.json`:

```json
{
  "scripts": {
    "lombok": "lombok-ts generate",
    "lombok:watch": "lombok-ts watch",
    "lombok:clean": "lombok-ts clean"
  }
}
```

There are four subcommands. All of them work today, except `watch` which is a stub for Phase 2.

## `lombok-ts init`

Drops a starter `lombok.config.ts` in the current directory.

```bash
npx lombok-ts init
```

Output:

```
Created lombok.config.ts
```

The file it writes:

```ts
import { defineConfig } from 'lombok-typescript';

export default defineConfig({
  backend: 'auto',
  codegen: {
    outputDir: '.lombok',
    include: ['src/**/*.ts'],
    exclude: ['node_modules', '**/*.test.ts', '**/*.spec.ts'],
    tsConfigPath: 'tsconfig.json',
    watch: false,
  },
});
```

If a `lombok.config.ts` already exists, the command refuses to overwrite. Pass `--force` to clobber:

```bash
npx lombok-ts init --force
```

## `lombok-ts generate`

The main command. Reads your config, walks every source file matching `codegen.include`, and writes a companion stub to `<outputDir>/<rel-source-path>.lombok.ts` for each file that has at least one class.

```bash
npx lombok-ts generate
```

Sample output:

```
Loaded config from /your/project/lombok.config.ts
Generated 3 companion files under .lombok/
  - /your/project/.lombok/src/user.lombok.ts (User)
  - /your/project/.lombok/src/order.lombok.ts (Order, OrderItem)
  - /your/project/.lombok/src/services/api.lombok.ts (ApiClient)
```

### What a companion file looks like today

For an input like:

```ts
// src/user.ts
@Data
@Builder
class User {
  @NonNull name: string = '';
  age?: number;
}
```

The output today is just a comment-form summary (Phase 1 will replace this with real generated TypeScript):

```ts
// .lombok/src/user.lombok.ts
// Auto-generated by lombok-typescript.
// Source: src/user.ts
// Do not edit. Regenerate via `lombok-ts generate`.

// Class: User
// Decorators: @Data, @Builder
// Fields:
//   - name: string [@NonNull]
//   - age: number?
// Methods:
//   (no methods)
export {};
```

### Flags

```
--output-dir <dir>   Override codegen.outputDir from the config
--ts-config <path>   Override codegen.tsConfigPath from the config
```

Example:

```bash
npx lombok-ts generate --output-dir generated --ts-config tsconfig.build.json
```

If no `lombok.config.*` is found, the CLI uses sensible defaults and tells you so:

```
No lombok.config.* file found, using defaults.
```

## `lombok-ts watch`

The plan is to re-run codegen on file change. Today this is a stub: it logs a message and exits. Phase 2 wires up the actual watcher.

```bash
$ npx lombok-ts watch
Watch mode is not implemented yet (Phase 2).
Re-run `lombok-ts generate` after changes.
```

For now, integrate `lombok-ts generate` into your existing watcher. With `tsup`, that's:

```json
{
  "scripts": {
    "build": "lombok-ts generate && tsup",
    "build:watch": "lombok-ts generate && tsup --watch"
  }
}
```

A pre-build step is fine for projects that don't change a lot of decorated classes during dev.

## `lombok-ts clean`

Removes generated and build artifacts from your project.

```bash
$ npx lombok-ts clean
Removed .lombok/
Removed dist/
Removed coverage/
```

By default it removes `.lombok/`, `dist/`, and `coverage/`. There's no flag to customize this from the CLI today; the underlying `runClean` function accepts a `paths` array, so if you want a different set, write a small script that calls it directly.

## Composing into your build

Three reasonable ways to wire this up:

1. **Pre-build step.** Run `lombok-ts generate` before `tsc` / `tsup` / `vite build`. Good for projects with mostly static class shapes.

2. **Pre-commit hook.** Run codegen before commit so `.lombok/` is always in sync. If you commit `.lombok/` to git this works; if you don't, prefer option 1.

3. **CI-only.** Run codegen in CI and ship the result. Local dev relies on whatever's checked in. Works if your team rarely changes decorated classes.

Pick whichever matches your team's workflow. There's no recommended default; the CLI is intentionally inert outside of explicit invocations.

## Programmatic use

The CLI is a thin wrapper. Anything it does, you can do from your own scripts:

```ts
import { CodeGenerator } from 'lombok-typescript/codegen';

const gen = new CodeGenerator({
  outputDir: '.lombok',
  include: ['src/**/*.ts'],
  exclude: ['node_modules', '**/*.test.ts'],
  tsConfigPath: 'tsconfig.json',
  watch: false,
});

const generated = await gen.generate();
console.log(`Generated ${generated.length} files`);
```

Useful when you want codegen to run as part of a custom build script or a Nx generator.
